# Food for Thought: Executables

Where do the executable files come from?

## FHS

::: note
If you skipped the food for thought chapter on the navigation previously, now would be a good time to revisit it: [the Food for Thought of the Navigation](02.1-navigation.md).
:::

The Filesystem Hierarchy Standard defines the default locations for system executables and the commands that should be available. It also makes recommendations about where executable files should not be stored. The following list contains some of the standard locations for executable files:

- [/bin](https://refspecs.linuxfoundation.org/FHS_3.0/fhs/ch03s04.html)
- [/sbin](https://refspecs.linuxfoundation.org/FHS_3.0/fhs/ch03s16.html)
- [/usr/bin](https://refspecs.linuxfoundation.org/FHS_3.0/fhs/ch04s04.html)
- [/usr/sbin](https://refspecs.linuxfoundation.org/FHS_3.0/fhs/ch04s10.html)
- [/opt](https://refspecs.linuxfoundation.org/FHS_3.0/fhs/ch03s13.html)

Many modern Linux distributions have centralised executables in directories under `/usr` and replaced the `/bin` and `/sbin` directories with links to the `/usr/bin` and `/usr/sbin` directories. The original locations are preserved for backwards compatibility.

## $PATH variable

As you may have noticed, you can simply type a command and the operating system will execute it. But how does the operating system find them?

The operating system searches executable files in directories defined by the `$PATH` variable. You can check your current `$PATH` variable using the command: `env | grep PATH`.

::: terminal
__HOSTNAME__@__USERNAME__:~$ env | grep PATH
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/usr/games:/usr/local/games:/snap/bin
__HOSTNAME__@__USERNAME__:~$ █
:::

The $PATH variable contains a list of directories to be searched, each with its absolute path. These paths are separated by colons. While it is possible to define relative paths, this is not recommended for security reasons.

For example, adding the relative path `./` to the `$PATH` variable could cause a possibly malicious executable called `ls` in your current location to be executed instead of the system's own version.

If you modify the `$PATH` variable, always add new entries to the end of the list, keeping system directories at the beginning. Directories are checked from left to right, and when a match is found, the corresponding file is executed.

## DIY

You can add your own executable files to the operating system. If you have administrative access, you can copy your files to directories in the $PATH variable. Alternatively, as a regular user, you can add your own directory (e.g. `~/bin`) to the $PATH variable.

This would mean that the operating system will also find your executable files without an absolute or relative path in front of the file name.

## man pages

So, what do the system executables do? Where can information on how to use the commands be found?

These days, we have AI to help us. However, perhaps your computer is offline and you have no internet access. Or sometimes, consulting the manual for a command can give you better ideas for achieving your goal than an LLM would suggest.

Unfortunately, the man pages are not installed in our learning environment on right. On many systems, though, you can type the command `man ls` to access a [comprehensive manual](https://man7.org/linux/man-pages/man1/ls.1.html) for the `ls` command.

To compensate for the lack of man pages in the learning environment, here's a link to an alphabetical list of commands and their man pages: [Linux manual pages](https://man7.org/linux/man-pages/dir_all_alphabetic.html).
