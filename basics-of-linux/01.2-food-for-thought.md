## Password login vs. SSH keys

Logging in to a remote server with a password is considered very insecure. Internet bots attempt to access Linux servers around the clock.

This is why almost none of the available services support password-only logins, instead using SSH keys.

Further information on using SSH keys in CSC environments can be found here:

- [Docs CSC: SSH Key Pair](https://docs.csc.fi/cloud/pouta/tutorials/ssh-key/)
- [CSC Computing Environment: Creating and adding SSH keys](https://csc-training.github.io/csc-env-eff/hands-on/connecting/ssh-keys.html)
- [Docs CSC: Setting up SSH keys](https://docs.csc.fi/computing/connecting/ssh-keys/)

::: service
However, for our exercise environment on right, a password login is sufficient, so there is no need to use SSH keys.
:::

## Terminals

The command line can be accessed via a terminal window. In Windows, this is done through PowerShell or a separate programme such as [PuTTY](https://putty.software/).

If you're interested in how terminals and terminal connections work, please follow this link:

- [How Terminals Work](https://how-terminals-work.vercel.app/)

::: note
Terminal connections can be local or made via a network (SSH). Terminologically local connections are made through _a console_.
:::

## Shell

The software you will use in these exercises and in the Linux terminal or console is a command-line shell.

One of the most popular shells is [Bash](http://www.gnu.org/software/bash/), which is what we use in our environment. There are others, too. For example, the default shell on macOS is [Z shell](https://www.zsh.org/).

You can install other shells in Linux. For example the mentioned Z shell (zsh) among others. A good list of available shells can be found on Wikipedia: [List of command-line interpreters](https://en.wikipedia.org/wiki/List_of_command-line_interpreters).

The used shell is usually not extremely important. The basic functionality of all of them is more or less the same. However, there may sometimes be subtle differences that are worth acknowledging. This is especially true when running scripts on your Linux machine.

::: note
POSIX compliant scripts must follow [Bourne shell](https://en.wikipedia.org/wiki/Bourne_shell) notation. Being POSIX compliant roughly means that the script can be run on almost all Linux and Unix machines.
:::
